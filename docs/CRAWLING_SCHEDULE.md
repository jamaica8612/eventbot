# CRAWLING_SCHEDULE

이 프로젝트의 핵심 흐름은 `자동 수집 -> DB 저장 -> 모바일 딸각 관리`입니다.

## 기본 원칙

- 크롤링은 웹앱 안에서 직접 돌리지 않습니다.
- 크롤러는 별도 작업으로 실행합니다.
- 웹앱은 저장된 이벤트를 읽고 상태를 관리합니다.
- 사이트에 부담을 주지 않도록 너무 자주 요청하지 않습니다.
- Cloudflare 확인 화면이나 보호 장치를 우회하지 않습니다.

## 1차 권장 간격

초기 운영은 하루 3회가 적당합니다.

```text
09:00
15:00
21:00
```

이유:

- 이벤트는 실시간 초단위 정보보다 놓치지 않는 것이 중요합니다.
- 사용자는 밤에 일하므로, 저녁/밤에 확인할 수 있는 최신 목록이 필요합니다.
- 무료/저비용 자동 실행 환경에서 부담이 적습니다.
- 슈퍼투데이 공개 AJAX 목록에 과한 요청을 보내지 않습니다.

## 확장 간격

운영해보고 부족하면 3시간 간격으로 늘립니다.

```text
09:00
12:00
15:00
18:00
21:00
00:00
```

1시간 간격은 당장은 사용하지 않습니다. 이벤트 관리 앱의 목적과 사이트 부담을 고려하면 초기에는 과합니다.

## 실행 방식 후보

### 1. 로컬 수동 실행

현재 가능:

```bash
npm run crawl:suto
```

장점:

- 가장 단순합니다.
- 비용이 없습니다.
- 크롤러 품질을 확인하기 좋습니다.

단점:

- 사용자가 직접 실행해야 합니다.

### 2. GitHub Actions 스케줄

현재 설정:

```text
하루 3회 자동 실행
크롤러 실행
DB에 upsert
```

`.github/workflows/crawl-suto.yml`에 UTC `00:00`, `06:00`, `12:00` 스케줄을 추가했습니다.
한국 시간 기준으로는 `09:00`, `15:00`, `21:00`입니다.

필요한 GitHub Secrets:

```text
VITE_SUPABASE_URL
SUPABASE_SERVICE_ROLE_KEY
```

Secrets가 비어 있으면 자동 크롤링 워크플로는 DB 저장 없이 성공 처리하지 않고 실패합니다.

장점:

- 무료로 시작하기 쉽습니다.
- PC를 켜두지 않아도 됩니다.

단점:

- Supabase 같은 외부 DB가 먼저 필요합니다.
- 비밀 키 설정이 필요합니다.

### 3. 서버 cron

나중 후보:

- 저렴한 VPS
- NAS
- 개인 PC 작업 스케줄러

초기에는 관리 부담이 있어서 우선순위가 낮습니다.

## 현재 결정

- Supabase Secrets가 없으면 `npm run crawl:suto` 수동 실행으로 JSON 하네스에 저장합니다.
- Supabase Secrets가 있으면 GitHub Actions 하루 3회 자동 실행으로 DB에 저장합니다.
- 자동 실행 전까지는 JSON을 임시 하네스로 유지합니다.
