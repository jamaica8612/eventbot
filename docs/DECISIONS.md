# DECISIONS

## Vite + React를 사용한다

- 초기 구조가 단순합니다.
- 컴포넌트 단위로 화면을 나누기 쉽습니다.
- 무료 또는 저비용 배포 선택지가 넓습니다.
- 사용자가 초보에 가까워도 Next.js보다 초반 개념 부담이 작습니다.

## 모바일 우선 반응형 웹앱으로 만든다

- 사용자는 밤에 택배 일을 하며 모바일로 짧게 확인해야 합니다.
- PC는 집에서 정리하고 천천히 참여하는 용도입니다.
- 모바일 앱과 PC 앱을 따로 만들면 유지보수 부담이 커집니다.
- 따라서 하나의 웹앱에서 같은 데이터와 상태값을 공유합니다.

## Wanted Design System을 시각 기준으로 삼는다

- 사용자가 디자인 품질을 중요하게 보고 Wanted Design System을 기준으로 요청했습니다.
- 현재 Figma 링크는 전용 컴포넌트/토큰을 직접 검색했을 때 결과가 비어 있어, 우선 Wanted 계열의 제품 UI 원칙을 코드 토큰으로 반영합니다.
- 기본 방향은 밝은 배경, 선명한 블루 액션, 얇은 보더, 8px radius, 절제된 카드 UI입니다.
- 폰트는 Pretendard를 사용합니다.

## 슈퍼투데이 크롤러는 웹앱과 분리한다

- 크롤러 코드는 `crawler/sutoCrawler.js`에 둡니다.
- 실행 명령은 `npm run crawl:suto`입니다.
- 수집 결과는 `public/crawled-events.json`에 저장합니다.
- 웹앱은 이 JSON을 읽어서 기존 목업 이벤트와 합칩니다.
- 현재는 슈퍼투데이 AJAX 인기 이벤트 목록만 수집합니다.
- `/cpevent?isActive=1` 목록 페이지와 상세 페이지는 Cloudflare 확인 화면이 나올 수 있어 우회하지 않습니다.

## JSON 저장은 임시 하네스다

- 최종 목표는 자동으로 이벤트를 가져와 DB에 저장하고 웹앱에서 관리하는 구조입니다.
- 지금 JSON을 쓰는 이유는 크롤러가 안정적으로 데이터를 가져오는지 먼저 확인하기 위해서입니다.
- 이벤트 데이터 형태가 안정되면 `크롤러 -> JSON -> 웹앱`을 `크롤러 -> DB -> 웹앱`으로 바꿉니다.
- 무료/저비용 운영과 나중 로그인 확장을 고려하면 1차 DB 후보는 Supabase입니다.

## 크롤링 간격은 하루 3회로 시작한다

- 크롤링은 웹앱 안에서 직접 돌리지 않습니다.
- 초기 권장 간격은 09:00, 15:00, 21:00 하루 3회입니다.
- 운영해보고 부족하면 3시간 간격으로 확장합니다.
- 1시간 간격은 초기에는 과하므로 사용하지 않습니다.
- 자세한 내용은 `docs/CRAWLING_SCHEDULE.md`에 기록합니다.

## 이벤트 상태는 localStorage에 먼저 저장한다

- 첫 저장 기능은 무료이고 설정이 필요 없는 브라우저 `localStorage`로 시작합니다.
- 기본/크롤링 이벤트는 이벤트 전체가 아니라 `eventId -> status`만 저장합니다.
- 목업 이벤트 내용은 `src/data/events.js`에 유지하고, 저장된 상태만 앱 시작 시 덮어씁니다.
- 나중에 DB를 붙일 때 `src/storage/eventStatusStorage.js`를 교체하거나 확장하면 됩니다.

## 직접 추가 이벤트는 현재 범위에서 제거한다

- 사용자의 핵심 목표는 직접 입력이 아니라 자동 수집입니다.
- 직접 추가 폼은 유지보수와 사용 흐름을 흐리므로 제거했습니다.
- 예외적으로 수동 보정이 필요해지면 나중에 별도 관리 기능으로 다시 설계합니다.

## AI 댓글 생성은 현재 범위에서 제외한다

- 이 프로젝트의 핵심은 댓글 자동 생성이 아닙니다.
- 핵심은 이벤트를 정리하고 빠르게 판단하고 참여 상태를 관리하는 것입니다.
- AI 댓글 생성은 기능 범위를 키우고 유지보수를 어렵게 만들 수 있습니다.

## 웹앱, 크롤러, 알림은 분리한다

- 웹앱은 이벤트 확인과 상태 관리에 집중합니다.
- 크롤러는 별도 폴더와 명령으로 관리합니다.
- 알림도 나중에 별도 작업으로 설계합니다.
- 한 파일에 섞으면 초보자가 유지보수하기 어렵습니다.

## 현재는 목업 데이터와 크롤링 JSON으로 시작한다

- 첫 단계에서는 DB보다 화면 흐름과 문서화가 중요합니다.
- 목업 데이터를 `src/data/events.js`에 분리해 나중에 저장 방식으로 교체하기 쉽게 둡니다.
- 크롤링 데이터는 `public/crawled-events.json`으로 시작합니다.

## `전체보기`는 미처리 이벤트 전체로 정의한다

- 사용자는 `전체보기` 상태에서 `나중에`, `참여함`, `제외`를 누르면 카드가 사라지는 흐름을 기대했다.
- 따라서 `전체보기`는 모든 상태를 무조건 보여주는 관리자식 전체가 아니다.
- 현재 앱에서 `전체보기`는 아직 판단하지 않은 `ready` 이벤트 전체다.
- 이미 처리한 이벤트는 `나중에`, `참여함`, `결과 미확인`, `당첨`, `미당첨`, `제외` 필터에서 다시 본다.

## 모바일 필터는 하단 고정 메뉴로 둔다

- 사용자는 리스트박스 방식이 불편하다고 피드백했다.
- 밤에 일하면서 한 손으로 누르는 흐름에는 셀렉트 박스보다 하단 고정 메뉴가 더 적합하다.
- 자주 쓰는 `전체`, `현장`, `나중`, `참여`, `결과`는 하단 메뉴에 둔다.
- 상대적으로 덜 자주 쓰는 `당첨`, `미당첨`, `제외`는 목록 상단 칩으로 둔다.
- 모바일과 PC는 같은 반응형 웹앱을 유지한다.

## 원문 상세 본문은 보호 페이지를 우회하지 않는다

- 슈퍼투데이 상세 페이지는 Cloudflare 확인 화면에 막힐 수 있다.
- 이 프로젝트는 공개적으로 접근 가능한 AJAX/HTML만 사용한다.
- 현재 카드는 목록에서 얻은 원문 제목, 링크, 플랫폼, 저장 수, 순위를 보여준다.
- 상세 본문 자동 수집은 우회가 필요하면 하지 않는다.

## 슈퍼투데이 최신/오늘마감 HTML 목록은 수집하지 않는다

- `cpevent` 일반 목록과 최신 정렬 URL은 현재 Cloudflare 확인 화면을 반환한다.
- 보호 화면 우회는 하지 않기로 한 원칙에 맞춰 최신/오늘마감 HTML 목록 수집은 보류한다.
- 대신 공개 AJAX 인기 목록의 `hot_cnt`를 50으로 늘려 더 많은 후보를 확보한다.
- 2026-05-02 확인 기준 인스타그램/출석류 제외 후 DB에 29개 이벤트가 저장됐다.

## 카드의 주 동작은 `참여하기`로 둔다

- 사용자는 `원문 보기`보다 슈퍼투데이 `응모하기` 링크로 바로 이동하는 흐름을 원했다.
- `원문 보기` 버튼은 제거하고 `참여하기` 버튼을 주 동작으로 둔다.
- 슈퍼투데이 응모 링크는 `bbs/link.php?bo_table=cpevent&wr_id={id}&no=1` 형식으로 만든다.
- 해당 링크가 Cloudflare나 로그인 상태에 영향을 받을 수 있으므로, 앱은 우회하지 않고 브라우저 이동만 담당한다.

## 당첨 화면은 장부형으로 관리한다

- 사용자는 슈퍼투데이의 당첨/가계부 화면처럼 당첨 내역을 모아보고 싶어 했다.
- `당첨`은 일반 이벤트 카드 목록이 아니라 날짜, 당첨내역, 금액, 수령상태를 보는 전용 화면으로 둔다.
- 모바일에서는 표를 억지로 넣지 않고 카드형 장부로 보여준다.
- PC에서는 같은 데이터를 표에 가까운 형태로 펼쳐서 보여준다.
- 현재는 DB 전 단계이므로 `prizeAmount`, `receiptStatus`를 localStorage에 저장한다.

## 화면은 설명보다 작업 중심으로 둔다

- 사용자는 현재 UI가 불편하고 관리하기 어렵다고 피드백했다.
- 큰 히어로 영역이나 설명형 화면보다 바로 처리할 상태가 먼저 보여야 한다.
- 상단은 `판단 대기`, `현장 딸각`, `결과 확인`, `당첨금`만 빠르게 보는 컴팩트 대시보드로 둔다.
- 하단 메뉴는 5개 이하로 유지한다. 너무 많은 하단 메뉴는 모바일에서 오히려 불편하다.
- `참여함`, `미당첨`, `제외`처럼 덜 자주 쓰는 보기는 보조 칩으로 관리한다.
